## Session 2: Phase 4.5c (cell breakdowns + visible grid)

### Goal

Address user feedback from first round of tester demos. Four testers reviewed
the live prototype at a7zain.github.io/sar-change-detection/. Build per-cell
category breakdowns with desert-masked stats and a visible, clickable grid
layer.

### User feedback (verbatim)

1. "the percentage of change seems overcalculated, if there's a way to provide
   more details of what actually changed and percentage of each. like is the
   street, new construction, water, vegetation, etc -- it's unclear"
2. "of course it's going to be better with a designated website/software, but
   as a prototype its sufficient. only it needs better resolution next"
3. "the zoom in-zoom out in webapp is little annoying, also hope there are more
   details options of detection like the stable green, lost green feature. We
   need more features next"
4. "overall this is very exciting"

### Decision: Phase 4.5c, not 4.5b or Phase 5

All four testers wanted more **depth** in Riyadh — classification breakdown,
categories, detail. Zero asked for more cities (not Phase 5). Nobody asked
"when did this change?" (not Phase 4.5b time-series). The signal pointed
clearly at Phase 4.5c: richer per-cell classification + UX polish.

### What shipped

#### Step 1: Recon

- Found grid export script: `src/prepare_webapp_data.py`
- Found classified rasters: `outputs/optical_change_2020_2026.tif` (EPSG:4326,
  1687x2002) and `outputs/phase4_green_map.tif` (EPSG:32638, 1687x2002)
- **Key finding:** same pixel dimensions despite different CRS — direct row/col
  indexing works, no reprojection needed
- Decided to skip SAR k=5 raster (`outputs/ml_classification.tif`) — different
  resolution (10m, 3374x4089), different extent, would require reprojection

#### Step 2: Data export (grid_index.json)

Modified `src/prepare_webapp_data.py` to compute per-cell breakdowns:
- `change_breakdown`: {new_construction, land_clearing, vegetation_change, stable} as %
- `vegetation_breakdown`: {stable_green, new_green, lost_green, other} as %
- `change_pct_masked`: desert-masked change percentage (suppresses false-positive
  changes in sandy areas)
- `valid_pixels`: count of non-nodata pixels per cell
- Included all 56 cells (removed old <10 changed pixels skip)

**Key methodology finding:** AOI-wide `change_pct` dropped from **42.6%**
(unmasked) to **17.8%** (desert-masked). The headline number was inflated by
desert speckle classified as "land clearing." Tester #1 was right — the
percentage was overcalculated.

**Desert mask bug caught and fixed:** First implementation removed desert pixels
from the denominator, which *inflated* the change ratio (stable pixels removed
from denominator). Correct approach: suppress desert-flagged changed pixels
(reclassify as stable), keep denominator constant.

#### Step 3: Popup template

Rewrote popup HTML in `webapp/index.html`:
- Headline: "Change in this area: X.X%" using `change_pct_masked`
- Subtext: "(raw, before desert filter: Y.Y%)" for methodology transparency
- "What changed" section: colored dots + mini bars + percentages, sorted
  descending, stable last in muted grey
- "Vegetation" section: conditional, only shown if any green category > 0.5%
- Before/after thumbnails: kept, with graceful onerror hiding
- Footer: "Cell {id} . {valid_pixels} valid pixels"

#### Step 4: Diagnosis — missing grid layer

The popup handler from step 3 was wired to `map.on('click')` — a global map
click listener. But the ROI GeoJSON layer (with `fillOpacity: 0.05`) intercepted
clicks inside ROI polygons via Leaflet's event propagation, preventing the map
click from firing. Additionally, there was no visual grid layer — users had no
affordance telling them "click here."

#### Step 5: Visible grid layer + ROI conflict fix

- Created `gridPane` (z-index 446, above roiPane at 445, below changePane at 450)
- Built 56 `L.rectangle` instances in an `L.layerGroup`, each with:
  - Fill opacity scaled by `change_pct_masked` (0%->0, 5%->0.10, 50%+->0.30)
  - Subtle orange stroke (weight 1, #ff8800, opacity 0.4)
  - Hover highlight (weight 2, opacity 0.8)
  - Click handler with `L.DomEvent.stopPropagation` to prevent ROI interference
- Extracted popup logic into reusable `buildCellPopup(cell)` function
- Removed old `map.on('click', handleMapClick)` handler
- Added ray-casting point-in-polygon for ROI cross-reference in popup footer
- Wired grid layer to "Show Changes" / "Hide Changes" toggle
- Added "Click a cell for breakdown" hint to change legend

### Process failures

#### (a) Uncommitted changes — time wasted on stale live site

Steps 2 and 3 both completed locally but were never committed/pushed before
testing on the live GitHub Pages site. Debugging time wasted verifying against
a stale deployment. Lesson: run `git status` after every Claude Code step
before trusting "done" messages.

#### (b) Popup wired to non-existent layer

Step 3 shipped a `map.on('click')` handler that appeared to work in isolation
but had no visual layer to click. The ROI layer silently consumed all clicks
inside its polygons. This was only caught during manual testing in step 4.

### What we learned

1. **Desert mask matters for stats, not just overlay.** The existing
   `prepare_webapp_data.py` applied the desert mask to the change overlay PNG
   but NOT to the `grid_index.json` stats. This caused the "overcalculated"
   perception.
2. **Always verify click propagation in multi-pane Leaflet apps.** Layers with
   low but nonzero `fillOpacity` are still interactive by default.
3. **`git status` is a session checkpoint.** Two silent-failure modes today:
   uncommitted changes + popup on non-existent layer. Both caught late.
