## The reframe

Walked into the session with two questions: what does Basira offer that
Google Earth / GEE don't, and should the project pivot toward live
Saudi-funded programs (e.g. Red Palm Weevil).

Outcomes:

- **Google Earth is not the real competitor.** Google Earth is a viewer, not
  a change-detection product. GEE is developer infrastructure requiring
  JavaScript/Python and is getting more quota-restricted in April 2026. The
  consumer-grade application layer for Saudi-specific change monitoring is
  still open.
- **RPW is the wrong vertical for Phase 1.** Satellite is a supporting, not
  primary, modality for RPW — the state of the art is in-situ acoustic
  sensors + UAVs. Basira would be a weak follower in a well-funded field.
- **Mega-project monitoring is the right Phase 1 scope.** Satellite is the
  primary tool. No government partnership required. Visually spectacular
  demos. Extends naturally to the longer arc.
- **The TPM follow-up frame reshapes everything.** Context updated: I met
  with the SARsatX CTO approximately one month ago regarding a Technical
  Project Manager role (not an EO specialist role). The follow-up should
  demonstrate scoping judgment, execution discipline, and clean
  communication — NOT attempt to compete with his EO scientists on
  methodological depth.

Phase 1 scope locked: Qiddiya (hero) + King Salman Park + Diriyah Gate, with
dust-robust SAR-optical fusion as the technical crown jewel. Deliverable is
a dashboard + 3-5 page technical memo PDF + outreach message.

## Key decisions (do not re-litigate)

- **TPM framing over EO-specialist framing.** The CTO is evaluating breadth,
  scoping, execution, communication — not methodological novelty. Every
  artifact is shaped for that reader.
- **Dual-track documentation.** Public CLAUDE.md has generic framing; the
  specific SARsatX context lives in gitignored .private/context.md.
  Privacy separation is deliberate.
- **Flat webapp data layout on main.** The per-city subfolder structure
  (phase4/riyadh/, phase4/jeddah/) is Phase 5 architecture for a Phase 5
  problem; it stays on the wip branch. Phase 1 uses the three-project
  scope, which is a different problem than multi-city.
- **Free imagery for Phase 1.** Sentinel-1 and Sentinel-2 only. Paid imagery
  (Planet, Maxar) is not part of Phase 1 and won't be suggested by Claude
  — making Sentinel look good is part of the story.
- **Two old README images removed, not replaced.** 05_change_hero.png and
  07_ml_classification.png told the old SAR-2022-2024 story. Keeping them
  would have anchored the repo to the old framing. Hero image slot stays
  open for a Qiddiya screenshot during Phase 1 build.

## New workflow (effective next session)

Drafted and approved a simplified workflow doc during this session. Key
changes from previous workflow:

- **No re-upload to Project knowledge every session.** GitHub connector
  handles it.
- **Session logs: I ask Claude to draft, I review and commit.** Lower
  activation energy than writing from scratch.
- **Master plan updates are rare**, not per-session. Master plan is for
  direction changes, not progress within the direction.
- **Explicit energy/state check-in at session start.** Tell Claude what I'm
  working on + how I'm feeling. Shapes how Claude responds for the rest of
  the session. Load-bearing — most past failure modes happened at tired
  moments.
- **Cleanup passes at objective boundaries only** (not session boundaries).
  This was the third such pass; it was the heaviest.
- **`.private/context.md` uploaded at session start only when the session
  is about the TPM deliverable.** Keep private context out of unrelated
  sessions.
